decide - Support Decision-making through Collaborative Evolution of Proposals

Björn Ebbinghaus

Goal

We aim to improve decision-making in groups by giving participants the tools to improve a set of proposals together, without the need of external influence, that puts the participants strictly in the role of just a voter.

To evaluate our ideas in the field we built a prototype you can find below.

Motivation

In an experiment in 2019 we noticed that involved participants in a decision not only want to express their own proposals, but also actively participate in the improvement of existing proposals. They used the means available to them to express these wishes and (mis)used the discussion function to propose improvements. The problem is that mere comments are not the same as proposing a new candidate to vote for.

We hope to move from commenting on why a proposal is not good or how it can be improved to creating new and improved proposals.

How does it work?

The idea on which we are building is similar to a version control system in software development, in which changes to a code base each have one (or more) predecessors on which they build up on. In our case a new proposal can have no, one or many predecessors (parents) and therefore fulfills one of these roles:

1. A proposal with no parents
A proposal can be independent. This is like the behavior in current systems were a proposal is made without any relationship to another proposal.

Examples: We should build a new playground. Or: We should improve our local parks.

2. A proposal with exactly one parent (Fork)
This represents a proposed change to an existing proposal.

Example: We should build a new playground with a slide.
Which is derived from We should build a new playground.

3. A proposal with more than one parent (Merge)
Proposals can also be derived from several predecessors. This can represent a merge of closely related proposals or even the formation of a coalition between rivaling proposals.

Example: We should build a new playground  in the park
Which is a merge of We should build a new playground and We should improve our local parks.

We hope to improve the quality and diversity of proposals by focusing on improving proposals. In the best case we are definitely trying to achieve — the given tools lead to a single, final merge. No need for a vote or similar.

Open research questions

  • How can we encourage evolution of proposals, while keeping the set of candidate small and relevant?

    Since we want to encourage change and therefore progress towards better decisions we naturally increase the total number of proposals that were made, by a far greater amount than a traditional system that allow proposals. As every proposal should be respected, we can not just hide a proposal which has a child that may or may not be an improvement for some. How can we determine a relevance for a given proposal?

  • How to vote on candidates with relations between them?

    Imagine you expressed your approval with a certain proposal. Later on you approve a child of that proposal. How should your vote be counted? Should we prioritize the vote for a newer proposal, because we assume new = better? This may also be important when making a decision with multiple possible winner, as in reality proposals that are very closely related are often not suited to be implemented at the same time. E.g. Add a slide to the playground and Add a jungle gym to the playground may be constricted by available space on the playground.

  • How can we help participants find candidates for a coalition?

    We are already experimenting with similar proposals. Similarity for now is based on the size of the shared voters. This can be useful (if not crucial) when trying to settle a decision in a unanimous agreement.
